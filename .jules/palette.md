## 2025-05-14 - Interactive Element Relationships: ARIA Controls
**Learning:** Found a common pattern where toggle buttons manage `aria-expanded` state but lack explicit relationships to the content they reveal. Without `aria-controls` on the button and a matching `id` plus `role="region"` with `aria-labelledby` on the content container, screen reader users might not know what content appeared or where it is.
**Action:** When building or fixing custom interactive elements like accordions or tabs, always explicitly link the trigger to the content block using `aria-controls` alongside an `id`, and provide a clear role (e.g., `role="region"`) and label for the content block using `aria-labelledby`.
## 2025-04-10 - Adding Focus States to Styled-JSX Components
**Learning:** In components using `<style jsx global>` (like `chat-widget.tsx`), Tailwind utilities like `focus-visible:ring-2` cannot be applied directly via class names in the JSX because they are either stripped or not processed correctly alongside standard CSS in that specific Next.js/styled-jsx context. Keyboard navigation accessibility relies heavily on these states.
**Action:** When adding or fixing accessibility focus states (`:focus-visible`) in Next.js components that rely heavily on `styled-jsx` for styling, always apply the focus styles using standard CSS properties (e.g., `outline` and `outline-offset`) within the `<style jsx>` block itself rather than adding Tailwind classes to the elements.
