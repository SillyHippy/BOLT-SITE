## 2024-04-19 - Keyboard Accessibility for Toggle Buttons
**Learning:** When using `<button>` elements to simulate radio groups or multi-step toggles (like in custom interactive calculators such as Cost Calculator and Service Time Estimator), default browser focus styles are often removed by Tailwind's `outline-none`. This removes critical keyboard accessibility and focus rings. Additionally, the lack of `aria-pressed` or `aria-selected` prevents screen readers from understanding which option is currently active.
**Action:** Always add explicit focus rings using `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-[color] focus-visible:ring-offset-2` to these custom toggles, and use `aria-pressed={state === id}` or equivalent ARIA attributes to indicate the selected state to assistive technologies.

## 2025-02-23 - Improve Async Button UX and Accessibility
**Learning:** Adding visual loading indicators to form submit buttons significantly improves the perceived performance and prevents accidental double-submissions, especially when dealing with external APIs or large file uploads (like the Google Apps Script upload in the footer form). Linking toggle buttons to their content panels using `aria-expanded` and `aria-controls` is crucial for screen reader users to understand the interface's state.
**Action:** When implementing async form submissions, always use a loading state with a visual indicator (like a spinner) and disable the button. When using buttons to toggle the visibility of other elements, always provide `aria-expanded` and `aria-controls` pointing to the `id` of the toggled element.
## 2024-05-12 - Accessibility for Custom Toggle Buttons
**Learning:** When using `<button>` elements to simulate radio groups or toggles, default Tailwind classes like `outline-none` can remove focus styles. In addition, these buttons need semantic grouping and state representation for screen readers.
**Action:** Always restore keyboard accessibility using explicit `focus-visible` ring classes. Additionally, use `aria-pressed` attributes on the buttons to indicate the selected state, and wrap the group in a container with `role="group"` and an `aria-label` to provide context to screen readers.
